Документ описывает архитектуру внутренней retrieval-платформы для корпоративной базы знаний. В первой части разбираются требования к ingestion pipeline: как забирать статьи из wiki, как выделять metadata, как нормализовать mixed RU/EN текст и как хранить версии документов для последующего переиндекса. Затем автор подробно описывает слой embeddings: рассматриваются multilingual sentence transformers, процесс батчевой генерации векторов, отдельный warmup на старте сервиса и требования к memory footprint при развёртывании в контейнере. Отдельный раздел посвящён API сервису на FastAPI: какие endpoint нужны для загрузки документов, как оформлять jobs и polling, почему public response должен быть отделён от internal diagnostics, и как в signals хранить readiness, timings и top-k category scores.

В технической части много внимания уделено retrieval pipeline. Сначала система выполняет lexical retrieval по BM25, затем поверх кандидатов запускается dense retrieval по embeddings, а финальный порядок документов уточняется reranker-моделью. Для длинных документов описывается chunk selection strategy: заголовок, introduction, conclusion и фрагменты с высокой term density получают дополнительный приоритет. Автор отдельно объясняет, почему нельзя полагаться только на первые два чанка, когда документ содержит архитектурное описание, benchmarking appendix и operational notes в разных местах. Есть раздел про observability: structured logs с job_id и document_id, latency metrics для preprocessing, categorization и tagging, readiness probe, а также отдельные счётчики ошибок при LLM postprocess.

В документе обсуждаются ограничения и компромиссы. Использование одного shared embedding provider уменьшает cold start и расход памяти, но требует аккуратного lifecycle management в приложении. Асинхронный HTTP слой не делает inference неблокирующим сам по себе, поэтому CPU-bound обработка переносится в threadpool. Описаны вопросы безопасности, rate limiting, retry policy и правила graceful degradation, если внешний provider недоступен. Большой блок посвящён данным: как хранить category catalog, каким образом добавлять bilingual prototypes, как не потерять английские technical terms в русских документах и как выстраивать versioned evaluation dataset без ссылок на локальные временные файлы.

Заключительная часть документа содержит roadmap по развитию платформы. Сначала предлагается стабилизировать API контракт и baseline benchmark, затем улучшить language-aware preprocessing, потом расширить long-document categorization на все чанки и только после этого переходить к иерархической taxonomy. Дополнительно обсуждаются практики code review, release checklist, container image hardening, миграции базы и стратегия по нагрузочному тестированию. В приложении приведены примеры SQLAlchemy моделей, схемы очереди задач, конфигурация health и readiness endpoint, а также сценарии ручной проверки semantic search quality на документах из внутренней базы знаний.
